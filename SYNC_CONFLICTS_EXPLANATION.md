# 双向同步冲突解释

## 问题
运行 `node scripts/obsidian-sync.js --direction both` 后，显示检测到 10 个冲突，这是什么原因？

## 根本原因分析

### 冲突类型：`target_is_newer`

所有 10 个冲突的原因都是 `target_is_newer`，这意味着：**目标文件比源文件新**。

### 具体场景

**场景 1：Obsidian → Blog 同步**
- 源文件：`obsidian-vault/生活日记/xxx.md`
- 目标文件：`blog/life/xxx.html`
- **如果 MD 文件比 HTML 文件新 → 执行同步**
- **如果 HTML 文件比 MD 文件新 → 跳过同步，记录冲突**

**场景 2：Blog → Obsidian 同步**
- 源文件：`blog/life/xxx.html`
- 目标文件：`obsidian-vault/生活日记/xxx.md`
- **如果 HTML 文件比 MD 文件新 → 执行同步**
- **如果 MD 文件比 HTML 文件新 → 跳过同步，记录冲突**

### 为什么会发生？

在你的案例中，这 10 个文件的 MD 版本（在 Obsidian 中）比 HTML 版本（在 Blog 中）新：

```
示例：规划认识（2026.04）
├─ HTML (源): 4/7/2026, 12:39:38 PM (11805 bytes)
└─ MD (目标): 4/11/2026, 8:44:00 AM (3178 bytes)
               └─ 比 HTML 新 4 天
               └─ 跳过同步，避免覆盖较新的 MD 文件
```

这是因为：

1. **初始状态**：这些文件最初从 Obsidian → Blog 同步（创建 HTML）
2. **反向同步**：之后运行 Blog → Obsidian 同步，创建/更新了 MD 文件
3. **时间差**：MD 文件在 4/11 被修改，而 HTML 文件仍然是 4/7 的版本
4. **保护机制**：系统检测到 MD 文件更新，跳过同步以避免覆盖

## 这是错误吗？

**不是！这是预期的行为。**

这些"冲突"实际上是系统的**智能保护机制**：

- ✅ **避免数据丢失**：防止较新的 MD 文件被旧的 HTML 文件覆盖
- ✅ **双向编辑安全**：你可以在 Obsidian 和 Blog 两边分别编辑
- ✅ **自动解决**：系统会自动选择最新的版本，无需手动干预

## 如何处理？

### 方案 1：忽略（推荐）

**什么都不用做**。这些冲突是正常的，不影响功能：

- 最新的文件（MD 版本）已经保留在 Obsidian 中
- 索引已正确更新
- 博客展示正常

### 方案 2：强制同步（如果你确定要覆盖）

如果你确定要用 HTML 版本覆盖 MD 版本，可以：

```bash
# 方法 1：删除 MD 文件，重新同步
rm /Users/liang/Documents/GitHub/xie-l.github.io/obsidian-vault/生活日记/规划认识（2026.04）.md
node scripts/obsidian-sync.js --direction blog-to-obsidian

# 方法 2：修改文件时间戳，让 HTML 看起来更新
touch /Users/liang/Documents/GitHub/xie-l.github.io/blog/life/规划认识（2026.04）.html
node scripts/obsidian-sync.js --direction blog-to-obsidian
```

### 方案 3：手动合并

如果你两边都修改了，需要手动合并内容：

1. 打开冲突的 MD 文件和 HTML 文件
2. 比较内容差异
3. 手动合并到 MD 文件中
4. 重新运行同步

## 最佳实践

### 1. 单向编辑（避免冲突）

如果可能，尽量只在一个地方编辑：

- **只在 Obsidian 编辑**：运行 `obsidian-to-blog` 同步
- **只在 Blog 编辑**：运行 `blog-to-obsidian` 同步

### 2. 定期双向同步（接受冲突）

如果两边都会编辑：

- 接受会有冲突的事实
- 定期运行双向同步
- 信任系统选择最新版本
- 必要时手动合并

### 3. 查看冲突日志

运行同步时，查看详细的跳过信息：

```bash
node scripts/obsidian-sync.js --direction both 2>&1 | grep "跳过同步"
```

输出示例：
```
[INFO] 目标文件已是最新，跳过同步: 规划认识（2026.04）.md 
      (目标: 4/11/2026, 8:44:00 AM, 源: 4/7/2026, 12:39:38 PM)
```

## 总结

**冲突不是错误，是保护机制！**

- ✅ `target_is_newer` = 目标文件比源文件新（正常）
- ✅ 系统会自动跳过，避免覆盖新内容
- ✅ 无需手动处理，除非确定要覆盖
- ✅ 不影响博客展示和功能

这些"冲突"实际上证明了同步系统在正确工作，保护你的数据不被意外覆盖。
